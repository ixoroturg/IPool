# IPOOL

## Документация:

IPOOL - реализация пула объектов. Состоит из 2 частей: пула и объекта пула

### Объект пула 
`IPoolEntry` - Обёртка для объекта из пула. Он представляет объект пула. Его поля:

- `value` - объект типа T, который представляет нужный вам объект пула
- `IPoolEntry reset()` - метод, который сбрасывает состояние объекта к первоначальным настройкам, возвращает себя же для цепочечного вызова
- `void close()` - наследуется от интерфейса `AutoCloseable`. Кладёт объект обратно в пул. У вас остаётся доступ к объекту, так что этот метод нужно вызывать только после завершения всех операций над ним

### Создание пула
`byte` константы `IPool`, чтобы решить, что делать при невалидном объекте пула:
- `RESET = 0` - сбросить объект в начальное состояние
- `RE_CREATE = 1` - пересоздать объект
- `CUSTOM = 2` - кастомная функция
- `EXCEPTION = 3` - бросить `RuntimeException`

Объект создаётся через `Buidler`. Его методы:

- `generator(Supplier<T>)` - задаёт генератор объектов для пула. Обязательный параметр!
- `size(int)` - задаёт начальный размер пула. По умолчанию стоит 8
- `maxSize(int)` - задаёт максимальный размер пула. Значение 0 отключает лимит. По умолчанию стоит 0
- `scale(float)` - задаёт множитель увеличения размера пула, если все объекты пула заняты. Пул всегда увеличивается минимум на 1. По умолчанию стоит `1.5`
- `reset(Function<T,T>)` - задаёт функцию сброса настроек. Она принимает объект пула и должна вернуть объект пула с настройками по умолчанию
- `validator(Predicate<T>)` - задаёт функцию для проверки валидности объекта пула перед его открытием. Если объект валиден, то эта функция должна вернуть `true`
- `onInvalid(byte)` - что делать, если `validator` вернул `false`. Принимает значение констант. По умолчанию `EXCEPTION`
- `maxRetry(int)` - максимальное количество попыток валидировать объект
- `fallback(byte)` - что делать, если `maxRetry` превышен. Принимает значение констант, но не может использовать `CUSTOM`. Сделан на случай, если `CUSTOM` оживитель объекта не сможет ничего сделать. По умолчанию `EXCEPTION`
- `reviver(Function<T,T>)` - функция для оживления объекта пула. Функиця принимает текущий объект пула, оживляет его и возвращает. Если функция вернёт `null`, то считается, что попытка воскрешения провалилась
- `destroy(Consumer<T>)` - данная функция закрывает объект пула, чтобы он мог освободить все свои ресурсы. Если установлена, то вызывается при `RE_CREATE` и при полном закрытии пула
- `autoReset()` - включает автоматический вызов `reset()` на обёртке объекта пула, чтобы всегда получать объект с начальными настройками. По умолчанию выключено
- `timeout(int ms)` - если размер пула составляет `maxSize` и все объекты заняты, то вызов `open()` заблокирует поток выполнения, пока объект не освободится. При истечении ms времени выбросит `RuntimeException`. Значение 0 отключает таймаут, что потенциально может привезти к вечной блокировке. По умолчанию стоит 0
- `build()` - возвращает настроенный пул, вызывается в самом конце цепочки вызовов `Builder`

### Пул

`IPool<T>` - пул. Он содержит следующие методы:

- `open()` - возвращает `IPoolEntry`. Блокирует вызов до освобождения объекта пула и  его валидации (если установлен `validator`)
- `async(Consumer<T>)` - асинхронный вызов в виртуальном потоке. Функция получит внутренний объект пула и самостоятельно вернёт его в пул после завершения функции
- `asyncReset(Consumer<T>)` - асинхронный вызов в виртуальном потоке. Функция получит внутренний объект пула, сброшенный в начальное состояние и самостоятельно вернёт его в пул после завершения функции
- `static <T>newBuilder()` - создаёт `Builder` для пула
- `close()` - наследуется от `AutoCloseable`. После его вызова любая попытка получить объект пула получит `RuntimeException` о закрытии пула. Так же вызовет `destroy` для всех объектов пула для полного освобождения ресурсов

### Логика открытия

`IPool` содержит массив объектов пула. Открытие объектов происходит синхронно

Алгоритм открытия объекта:

1. пробежаться по массиву. Если есть `null` объекты, то создаётся и возвращается новый объект пула. Если находится закрытый объект пула, то он открывается и возвращается
2. объект пула не нашёлся, то если есть закрытые объекты (если объект разблокировался после того, как цикл его проверил), то начать открытие заново
3. если размер пула не превышает его максимально допустимый размер, то пул увеличивается и процесс открытия начинается заново
4. если размер пула равен его максимально допустимому размеру, то выполнение блокируется до истечения таймаута
5. после успешного открытия объекта он метод открытия открывается для других потоков, а на открытом объекте проверяется его валидность
6. если объект не валиден, то выполняет действие, согласно `onInvalid` настройке пула, после чего валидация проходит ещё раз
7. если количество попыток достигло `maxRetry`, то используется действие согласно `fallback` и в случае неудачи выбрасывает `RuntimeException`. Счётчик попыток сбрасывает только при удачном возврате объекта и сохраняется между вызовами `open()` в случае провала
8. при удачной валидации возвращает открытый объект для работы
9. `async() и asyncReset()` вместо `RuntimeException` получают `null` в качестве своего аргумента
